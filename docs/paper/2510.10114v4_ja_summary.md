# LinearRAG 論文詳細要約

対象: `docs/paper/2510.10114v4.pdf`  
原題: `LINEARRAG: LINEAR GRAPH RETRIEVAL-AUGMENTED GENERATION ON LARGE-SCALE CORPORA`

この文書は論文の日本語による詳細要約です。全文翻訳ではなく、各セクションの主張、手法、実験設定、結果、付録の補足までをできるだけ落とさず整理しています。

## 0. 論文全体の要旨

この論文は、既存の GraphRAG が依存している「関係抽出ベースの知識グラフ構築」が、精度・一貫性・コストの面で大きなボトルネックになっていると指摘する。著者らは、グラフ検索の強みそのものを捨てるのではなく、関係抽出を捨てるべきだと再設計し、`LinearRAG` を提案する。

LinearRAG の中心的な考えは次の 2 点である。

- 文書間に散らばった情報をつなぐ主要なアンカーは「関係」よりも「エンティティ整合」である
- 関係の意味は三つ組に圧縮するより原文パッセージ中に保持した方が損失が少ない

そのため、LinearRAG は `entity`、`sentence`、`passage` の 3 種類のノードからなる `Tri-Graph` を構築し、関係ラベル付きエッジを持たない。検索は二段階で行う。

- 第 1 段階: sentence-entity 側で意味的伝播を行い、関連エンティティを活性化する
- 第 2 段階: passage-entity 側で Personalized PageRank を行い、重要パッセージを回収する

この構成により、グラフ構築時の LLM トークン消費をゼロにしつつ、複雑な multi-hop 質問でも高精度な検索と回答生成を達成したと主張している。

## 1. Abstract

Abstract では、従来 RAG は単純な質問には有効でも、大規模で非構造なコーパスでは情報が断片化しており弱いと整理している。GraphRAG は関係構造を導入して multi-hop 推論を強化しようとするが、既存手法はグラフ構築時の relation extraction に依存し、その工程が高コストかつ不安定で、ノイズの多いグラフを生み、結果的に検索品質を劣化させる。

提案法 LinearRAG は relation-free な階層グラフ `Tri-Graph` を、軽量な entity extraction と semantic linking のみで構築する。これにより以下を実現すると述べる。

- コーパスサイズに対して線形にスケールする構築
- 追加の LLM トークン消費なし
- 原文パッセージをそのまま知識キャリアとして保持する損失の少ない索引化

検索では、局所的な semantic bridging による relevant entity activation と、大域的 importance aggregation による passage retrieval の二段階を採用する。4 つのデータセットでベースラインを大きく上回ったとまとめている。

## 2. Introduction

導入部では、RAG が外部知識により LLM の幻覚を抑える有力な手法である一方、大規模非構造コーパスでは関連情報が複数文書に分散し、単純なチャンク検索では文脈のつながりを落としてしまうと述べる。GraphRAG はこの問題に対して、知識の階層性や依存関係を明示的グラフで表現し、粗粒度から細粒度へ検索する発想を取る。

ただし著者らは、近年の評価では GraphRAG が実運用に近い設定で naive RAG を下回ることが少なくないと指摘する。その主因は、自動構築された知識グラフの質の低さにあるという立場を取る。具体的には次の 2 種類の欠陥を強調している。

- 局所的不正確性: relation extraction が誤った三つ組を生成し、事実を壊す
- 大域的不整合性: 文書ごとに独立抽出された関係が、コーパス全体で整合しない

この問題意識から、著者らは GraphRAG のパイプラインを見直し、複雑な relation graph を「エンティティと原文パッセージを結ぶ線形で索引しやすい構造」に単純化する方針を打ち出す。導入部での主要貢献は次の通り。

- relation extraction 依存の GraphRAG の弱点を定式化
- relation-free な Tri-Graph 構築を提案
- semantic bridging と global importance aggregation からなる二段階検索を提案
- 4 データセットで精度・効率・スケーラビリティを実証

## 3. Preliminary Study

### 3.1 GraphRAG の性能劣化

事前調査では GraphRAG-Bench 上で naive RAG と代表的 GraphRAG を比較し、GraphRAG は evidence recall を上げる一方で context relevance を大きく落とすと報告する。論文中の例では、Vanilla RAG の context relevance は 62.87% であるのに対し、GraphRAG 系は 36.86% から 54.61% に留まる。

著者らの解釈は、「グラフ化により広く拾えるが、誤った接続も一緒に拾ってしまう」というものだ。たとえば climate change impacts を問うとき、弱いグラフ接続に引きずられて economic policies の文脈まで拾うようなズレが生じる、と説明している。

### 3.2 グラフ品質のエラー分析

性能悪化の原因を確かめるため、知識グラフ構築の誤りを細かく分析している。代表例は次の 2 種類である。

- 局所的不正確性: `Einstein did not win the Nobel Prize for his theory of relativity` が `(Einstein, won Nobel Prize for, theory of relativity)` のように否定を落として誤抽出される
- 大域的不整合性: `AI` と `Unsupervised Learning`、`NLP`、`CV` が平列な下位カテゴリとして雑に接続され、本来の階層構造を反映しない

著者らは、このような誤りが retrieval 時に semantic noise を増やし、正しい reasoning chain ではなく不整合な近傍をたどらせると結論づける。

### 3.3 Discussion

この節では、従来 GraphRAG の「明示的 relation extraction + triple knowledge representation」に対する根本的な批判をまとめている。

- 第一に、自然言語中の関係は複雑で、単純な atomic triple に落とすと意味が欠落しやすい
- 第二に、そもそも explicit relation extraction は不要で、passage をまたいで情報をつなぐ主役は aligned entities である

たとえば `Rachel reluctantly agreed to go running with Phoebe` のような文は、単一 triple に落とすとニュアンスを保ちにくい。著者らは、関係の意味は元テキストに保持し、推論時に LLM がそれを読む方が自然だと主張する。

## 4. The Framework of LinearRAG

### 4.1 全体像

LinearRAG は 3 種類のノードを持つ。

- entity nodes
- sentence nodes
- passage nodes

エッジは次の 2 種類だけである。

- entity と sentence の接続
- entity と passage の接続

つまり、relation label を持つ entity-entity グラフを作らない。これにより、関係抽出由来の誤差を根本から避ける。検索は 2 段階で進む。

- 段階 1: sentence-entity 部分グラフ上で relevant entity activation
- 段階 2: passage-entity 部分グラフ上で Personalized PageRank による passage retrieval

著者らは、これが graph construction と retrieval の両方で線形スケーラビリティを持つと述べる。

### 4.2 3.1 Token-Free Graph Construction

この節では `Tri-Graph` の構築方法を定義している。コーパス中の各 passage を文単位に分割して sentence 集合 `S` を作り、軽量な NER モデルで entity 集合 `E` を抽出する。ノード集合は次の 3 つ。

- `V^p`: passages
- `V^s`: sentences
- `V^e`: entities

接続規則は単純である。

- passage `p_i` が entity `e_j` を含めば edge `(V^p_i, V^e_j)` を張る
- sentence `s_i` が entity `e_j` を含めば edge `(V^s_i, V^e_j)` を張る

これを 2 つの隣接行列で表す。

- `C`: passage-entity の contain matrix
- `M`: sentence-entity の mention matrix

重要な主張は次の通り。

- 新しい passage が増えたときは、その passage だけを分割・NER・接続すればよい
- OpenIE より NER の方が軽く正確で、LLM トークンコストが不要
- `C` と `M` は疎行列として持てるので、メモリも線形で抑えやすい
- 原文 passage を保持しているため情報損失が少ない

つまり、グラフは relation summary ではなく、原文への「構造化インデックス」として機能する。

### 4.3 3.2.1 Relevant Entity Activation via Semantic Bridging

第一段階の目的は、クエリに直接出てくる entity だけでなく、multi-hop 推論に必要な中間 entity も活性化することにある。単純な entity match では bridging entity を見逃すため、意味的伝播を行う。

処理は概ね以下である。

1. クエリから spaCy で entity を抽出する
2. 各クエリエンティティに対し、知識グラフ内で最も類似した entity を見つけ、初期活性値を与える
3. クエリと各 sentence の類似度 `σ_q` を計算する
4. sentence-entity 二部グラフ上で、sentence relevance を重みとして entity activation を反復更新する

更新式は、前反復の活性ベクトルと sentence relevance を組み合わせ、隣接行列 `M` を通じて伝播させる形になっている。著者らはこれを、GraphRAG における relation matching に相当するが、明示的 relation graph を使わない implicit relation matching だと位置づける。

さらに、探索空間の爆発を防ぐため `dynamic pruning` を導入している。

- 新たに活性化した entity のスコアが閾値 `δ` を超えた場合だけ次反復に残す
- 新規 entity が閾値を超えなくなった時点で自動停止する

これにより、クエリごとの複雑さに応じて propagation 深さが自動調整される。

### 4.4 3.2.2 Passage Retrieval via Global Importance Aggregation

第二段階では、第一段階で得た activated entities を種として、passage-entity 二部グラフ上で Personalized PageRank を走らせる。目的は、局所的に関連する entity から見て、大域的に重要な passage を選ぶことである。

entity ノードの初期スコアは第一段階の活性値をそのまま使う。一方、passage ノードの初期スコアは次の 2 要素を組み合わせる。

- クエリと passage の DPR 類似度
- 活性化 entity の寄与を集約した項

entity 側の寄与では、該当 entity の passage 内出現回数や階層レベルを使って重み付けしている。最後に PPR スコアで上位 `k` passage を返す。

この節のポイントは、第一段階が「どの entity を起点に考えるべきか」を決め、第二段階が「その起点から見てグローバルに重要な passage は何か」を決めるという役割分担である。

## 5. Experiments

### 5.1 実験設定

評価対象データセットは 4 つである。

- HotpotQA
- 2WikiMultiHopQA
- MuSiQue
- Medical

前 3 つは multi-hop QA ベンチマークで、Medical は GraphRAG-Bench 由来のドメイン特化データセットである。各検証セットから 1,000 問を使い、同一コーパス上で比較している。

ベースラインは次の 3 群に分けられている。

- Zero-shot LLM: Llama 8B, Llama 13B, GPT-3.5-turbo, GPT-4o-mini
- Vanilla RAG: Top-1, Top-3, Top-5
- GraphRAG: KGP, G-Retriever, RAPTOR, E2GraphRAG, LightRAG, HippoRAG, GFM-RAG, HippoRAG2

評価指標は以下。

- `Contain-Acc.`: 正解文字列が出力に含まれるか
- `GPT-Acc.`: LLM ベースの正誤評価
- `Context Relevance`: 検索コンテキストの質問整合性
- `Evidence Recall`: 正答に必要な証拠を回収できたか

全手法で埋め込みモデルは `all-mpnet-base-v2`、生成と評価の LLM は `GPT-4o-mini`、検索件数は `k=5` に統一している。

### 5.2 4.2 Generation Accuracy (Q1)

Table 1 の結果では、LinearRAG が全 4 データセットで最良性能を示す。

主な数値は次の通り。

- HotpotQA: `Contain-Acc. 64.30`, `GPT-Acc. 66.50`
- 2Wiki: `Contain-Acc. 70.20`, `GPT-Acc. 63.70`
- MuSiQue: `Contain-Acc. 33.90`, `GPT-Acc. 37.00`
- Medical: `GPT-Acc. 63.72`

著者らの観測は 3 点ある。

- `Obs.1`: Zero-shot LLM に比べ、RAG 導入で精度は大きく上がる
- `Obs.2`: GraphRAG は vanilla RAG が落としやすい multi-hop 依存を補える
- `Obs.3`: ただし relation-extraction 依存の GraphRAG はグラフ品質に敏感で、LinearRAG がそれを上回る

特に 2Wiki では、LinearRAG の `GPT-Acc. 63.70` は次点より約 3.8 ポイント高いと述べている。

### 5.3 4.3 Efficiency Analysis (Q2)

Table 2 では、indexing 時間、retrieval 時間、LLM token 消費、平均精度を比較している。LinearRAG の値は以下。

- Indexing: `249.78 s`
- Retrieval 平均: `0.093 s`
- Prompt tokens: `0`
- Completion tokens: `0`
- Accuracy: `66.95`

比較対象の例としては以下。

- LightRAG: indexing `4933.22 s`, retrieval `10.963 s`, prompt `35.52M`, completion `51.16M`, accuracy `47.10`
- HippoRAG: indexing `936.00 s`, retrieval `1.461 s`, token 消費あり, accuracy `63.00`
- GFM-RAG: indexing `1202.77 s`, retrieval `1.211 s`, accuracy `63.20`

著者らの解釈は次の通り。

- `Obs.4`: 複雑な graph construction は index/retrieval の両方で重い
- `Obs.5`: LLM token を大量投入しても性能改善は保証されない
- `Obs.6`: LinearRAG は速度・コスト・精度の総合バランスが最も良い

ここで重要なのは、E2GraphRAG や RAPTOR は retrieval 自体は速くても、性能面で LinearRAG に劣る点である。

### 5.4 4.4 Ablation Study (Q3)

アブレーションでは 2 つの主要モジュールを外して比較している。

- `w/o Entity Activation`: semantic bridging を使わず、クエリから直接抽出した初期 entity のみ利用
- `w/o Global Importance Aggregation`: PPR を使わず、第一段階の初期スコアで直接文書回収

結果として、どちらを外しても 4 データセットすべてで性能が低下した。著者らは次のように整理する。

- 第一段階は、hidden logical relationship を持つ中間 entity を見つける役割
- 第二段階は、活性化されたサブグラフ上で文書重要度を大域的に見積もる役割

両者は補完的であり、片方だけでは LinearRAG の性能は成立しないと結論づける。

## 6. Conclusion

結論では、LinearRAG を「高価で誤りやすい relation extraction を軽量な entity extraction に置き換えた新しい GraphRAG」と総括している。Tri-Graph と二段階検索により、precision-recall のトレードオフを改善し、検索精度、生成精度、スケーラビリティのすべてで既存法を上回ると主張する。

## 11. Appendix D: 線形スケーラビリティ解析

この付録は、LinearRAG が graph construction と retrieval の両方で線形スケーラビリティを持つことを理論的に説明している。

### 11.1 構築段階

文分割と NER が中心で、計算量は `O(|P| · T)`。`|P|` は passage 数、`T` は平均 passage 長である。LLM トークン消費はゼロ。

メモリは主に以下からなる。

- passage, node, sentence embeddings: `O(|P| · T)`
- 疎行列 `M`, `C`: 実質 `O(|P| + |S|)` で、`|S|` は `|P|` に比例

したがって全体として `O(|P| · T)` に収まるとする。

### 11.2 検索段階

検索では similarity computation、SpMM による semantic propagation、PPR 反復を行う。各伝播ステップは疎行列の非ゼロ要素数 `nnz` に比例し、PPR も辺数に線形で計算できるため、全体計算量は `O(|P|)` としている。

さらに、行列化された実装のため並列計算との相性が良く、理論だけでなく実験でも高速化が確認できたと述べる。

## 12. Appendix E: Additional Experiments

### 12.1 E.1 Retrieval Quality Evaluation (Q4)

Medical データセットで、fact retrieval、complex reasoning、contextual summarization、creative generation の 4 タスクに対して recall と relevance を比較している。LinearRAG の成績は次の通り。

- Fact Retrieval: `Recall 88.86`, `Relevance 86.09`
- Complex Reasoning: `Recall 87.03`, `Relevance 81.58`
- Contextual: `Recall 89.13`, `Relevance 87.89`
- Creative Generation: `Recall 89.08`, `Relevance 72.74`

著者らの観測は次の 2 点である。

- `Obs.8`: 一般的 GraphRAG は recall を上げやすいが relevance を大きく落としがち
- `Obs.9`: LinearRAG は high recall と high relevance を両立する

例として creative generation では、GFM-RAG は recall を `44.88` から `83.51` に上げる一方、relevance は `58.73` から `22.87` に落ちる。これに対し LinearRAG は high recall を保ったまま relevance も高い。

### 12.2 E.2 Hyper-parameter Sensitivity (Q5)

2WikiMultiHopQA 上で `δ` と `λ` を調べている。

- `δ`: dynamic pruning の閾値
- `λ`: passage 初期スコアにおける DPR 類似度と entity 情報のトレードオフ係数

観測結果は次の通り。

- `Obs.10`: `δ` が小さすぎるとノイズが増えて効率が悪化し、大きすぎると関連 entity を取り込めない。実験では `δ = 4` を採用
- `Obs.11`: `λ` は小さめがよく、例として `0.05` 付近が最良。これは entity 情報が主成分で、DPR 類似度は補助信号でよいことを示す

### 12.3 E.3 Different Sentence Embeddings (Q6)

- all-mpnet-base-v2
- all-MiniLM-L6-v2

結果として大差はなく、LinearRAG は埋め込みモデル選択に比較的ロバストとされる。とはいえ、多くのデータセットで `all-mpnet-base-v2` が安定して良く、計算効率も踏まえてデフォルト採用している。

### 12.5 E.5 Case Study (Q8)

2WikiMultihopQA の質問 `Beatrice I, Countess of Burgundy の夫の国籍は何か` に対するケーススタディを示している。正解は `Germany`。

HippoRAG2 は明示的 relation に依存しており、`husband` に対応する事前抽出関係がうまく働かず、無関係文書を複数拾って `French` と誤答した。

一方 LinearRAG は以下の連鎖で中間 entity を活性化した。

- Iter 0: `Beatrice I`
- Iter 1: `Frederick Barbarossa`
- Iter 2: `Germany`

このように implicit relation を entity activation と context chaining でたどり、正答できた。著者らはこれを `Obs.14` として、明示的 relational tuple に依存しない利点の具体例にしている。

## 13. Related Work

関連研究は大きく 3 系統に整理されている。

- クラスタリングベースの階層構築
- relation-extraction ベースの知識グラフ構築
- reasoning-enhanced RAG

### 13.1 クラスタリングベース階層構築

Louvain や Leiden などの community detection で entity クラスタを見つけ、topic-based community に抽象化する流れである。長所はマクロな視点を与えることだが、誤った entity 関係が上位抽象で増幅されやすく、大規模グラフではスケーラビリティも悪いと批判している。

### 13.2 relation-extraction ベース GraphRAG

passage ごとに triple を抽出し、entity alignment で統合グラフを作る立場である。構造化された reasoning を適用しやすい一方、OpenIE が文脈窓制約の中で個別 passage に対して独立に行われるため、三つ組が不整合になりやすい。schema-guided な改善案もあるが、専門家による schema 保守が必要で汎化しにくいとする。

### 13.3 reasoning-enhanced RAG

こちらは明示的グラフを作るのではなく、LLM の推論能力で complex query を subquery に分解し、順に dense retrieval を行う流れである。LogicRAG、LAG、Chain-of-Note、SelfRAG などを挙げ、複雑質問への対応という目的自体は共有しつつ、LinearRAG は「グラフを作るが relation は作らない」という別の設計を取る点を際立たせている。

## 14. LLM Usage Statement

最後に、LLM は論文本文と付録の英文 polishing のみに使用し、研究アイデア、結果、主張の生成には使っていないと明記している。科学的貢献と実装はオリジナルだという立場である。

## 15. この論文の実務的含意

この論文の実務上の含意は明快である。GraphRAG を導入したいが、relation extraction の品質とコストに悩まされているケースでは、最初から relation graph を捨て、entity-centered な構造に寄せた方がよい可能性が高い。特に次の条件で相性が良い。

- multi-hop な依存は必要だが、relation extraction の誤りを許容できない
- 大規模コーパスを低コストで継続更新したい
- 検索時に原文文脈を損なわず、かつ graph traversal 的な広がりも欲しい

逆に言えば、厳密な関係ラベル自体が下流で必要なタスクでは、この設計だけで十分かは別途検証が必要である。
