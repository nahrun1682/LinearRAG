GraphRAG の強みは残しつつ、明示的な関係抽出だけをやめる。 その代わりに、エンティティ・文・パッセージ をつなぐ軽量グラフを作り、 意味伝播と PageRank で multi-hop 検索を実現する。

1. まず何が問題だったのか

論文の出発点は、GraphRAG は理屈の上では強そうなのに、実際には naive RAG を下回ることがあるという観察です。 著者たちは原因を「relation extraction ベースのグラフ構築」に絞って分析しています。

従来 GraphRAG の期待

文書をグラフ化すると、離れた文書同士のつながりをたどれるため、 複雑な質問でも必要な証拠を拾いやすくなるはずです。

実際に起きること

つながりを増やした分だけ、誤った関係や整合しない関係も増えます。 その結果、recall は上がっても relevance が落ち、回答生成が不安定になります。

局所的不正確性 単文からの relation extraction が否定やニュアンスを落とし、事実と異なる三つ組を生成する。
大域的不整合性 passage ごとに別々に抽出した関係が、コーパス全体で見ると矛盾・重複・構造崩れを起こす。
論文の発想転換 「関係を正しく抽出する」のではなく、関係を無理に抽出しない方向へ設計を切り替える。
重要なのは「情報をつなぐアンカーは relation ではなく entity で十分ではないか」という発想です。 関係の意味は三つ組より原文パッセージのほうが自然に保持できます。

2. LinearRAG のコアアイデア

LinearRAG は、relation graph を作らず、 entity / sentence / passage の 3 層だけで knowledge structure を作るのが最大の特徴です。

Tri-Graph のイメージ

Sentence A
Sentence B
Sentence C
↔
mention
Entity X
Entity Y
Entity Z
↔
contain
Passage 1
Passage 2
Passage 3

ポイントは、entity 同士を relation label で直接結ばないことです。 構造は軽く保ち、意味の本体は元のテキストに残します。

捨てたもの

OpenIE 的な relation extraction、relation label の品質管理、複雑な関係スキーマ。

残したもの

entity の整合、文脈の近さ、元の passage の意味情報、graph-based retrieval の広がり。

得たもの

線形スケーラビリティ、LLM トークン消費ゼロの indexing、ノイズに強い multi-hop 検索。

4. 具体例で理解するグラフ構築とRAG

ここでは、架空の3つのpassageをLinearRAGへ登録し、 「アーク研究所を設立した人物が受賞した賞は？」という質問に答えるまでを追います。 答えは1つのpassageには書かれておらず、人物を橋にして2つのpassageをまたぐ必要があります。

アーク研究所は、 量子材料を研究する独立機関である。 同研究所は佐藤葵によって設立された。

物理学者の佐藤葵は、 2024年に星雲科学賞を受賞した。 受賞理由は量子材料の研究である。

星雲大学は、 宇宙物理学と計算科学の教育で知られている。 研究所とは直接関係しない背景文書である。

1

seed entity を取る get_seed_entities()

質問NERが lothair ii を抽出 → embedding でグラフ内の最も近い entity に対応づけ（①の緑ノード）。

2

文を橋に選ぶ calculate_entity_scores()

lothair ii を含む文（①の紫の菱形）を質問と類似度比較し、上位 top_k_sentence 件だけ通す。

次の entity を活性化

選ばれた文に共起する別 entity（例: lotharingia や日付 11 november 875 など、①の黄色線の先）へ entity_score × 文類似度 を伝播。閾値未満は枝刈り。

PPR で順位付け run_ppr()

②のグラフ上で Personalized PageRank を実行し、スコアを構造伝播 → passage を降順に並べて上位 retrieval_top_k 件。

上位 passage を gpt-4o-mini に渡し、Thought: … Answer: … 形式で母の死亡年を生成。Answer: 以降が predictions.json に入る。

図と仕組みの対応

緑ノード= seed entity（質問から特定）
黄エッジ= entity↔sentence。multi-hop が伝わる経路
ピンク= 複数 passage を橋渡しする entity
灰エッジ= entity↔passage。passage 採点に効く
青四角= passage。PPR で最終順位が決まる
緑破線= 隣接 passage。局所的連続性を伝播

この test10 実行では LLM Accuracy 0.70 / Contain Accuracy 0.80（10問）でした。

要点：グラフは答えを作りません。「答えに必要な原文 passage を揃える検索器」として働き、 最終的な関係の解釈（「mother」「died」の対応づけ）は原文と LLM に委ねます。 だから relation ラベルを索引時に作らなくても multi-hop に答えられます。

6. 質問から回答まで、どう動くのか

パイプラインは大きく オフライン構築 と オンライン検索 に分かれます。 オフラインでは軽く索引を作り、オンラインでは 2 段階で relevant context を掘り当てます。

01

Graph Construction

passage を sentence に分割し、spaCy などで entity を抽出。sentence-entity と passage-entity の疎行列だけを持つ。

02

Entity Activation

クエリから entity を取り、sentence relevance を伝播させて、直接一致しない中間 entity まで活性化する。

03

Passage Ranking

活性化 entity を種に Personalized PageRank を回し、グローバルに重要な passage を上位から返す。

Stage 1: Semantic Bridging

直接の entity match だけでは multi-hop の橋渡し entity を取りこぼします。 そこで query と sentence の意味類似度を使い、sentence-entity グラフ上で関連 entity を少しずつ広げます。 ただし無限に広がるとノイズが増えるので、閾値 δ による dynamic pruning で拡散を止めます。

Stage 2: Global Importance Aggregation

活性化された entity から見て、どの passage が全体として重要かを Personalized PageRank で評価します。 局所一致だけでなく、周辺の構造全体を見て rank するので、単純な dense retrieval より reasoning chain を拾いやすくなります。

オンライン検索の実行シーケンス

1

質問を埋め込む

retrieve() が質問ベクトルを生成。後続のsentence・passage類似度で共用する。

2

Seed entityを決める

question_ner() で質問entityを抽出し、entity embeddingとの内積が最大の候補へ対応づける。

3

Sentenceを橋にentityを伝播

各active entityにつながる文のうち、質問に近い上位 top_k_sentence 件だけを通す。

4

Passage初期重みを作る

dense類似度とactive entityの出現ボーナスを合成し、PPRのreset vectorへ加える。

5

PPRで上位passageを返す

igraph.personalized_pagerank() のpassage部分だけを降順に並べ、上位k件を取得。

実装上のスコア分解

Entity propagation
next_entity_score
= current_entity_score
× cosine(question, sentence)

Passage reset weight
passage_score
= passage_ratio × normalized_DPR
+ log(1 + entity_bonus)

entity_bonus
= entity_score × log(1 + occurrences) / tier

tier が深いentityほど寄与を割り引き、同じentityがpassage内に複数回現れると対数的に加点します。 最後に passage_node_weight を掛けて、entity重みと同じreset vectorへ載せます。

default: BFS iteration

逐次探索版

  • active entityごとに未使用sentenceを取得
  • 質問類似度の高いsentenceだけを選択
  • 閾値未満のentityをその場で枝刈り
  • 処理が追いやすく、CPUでも動作可能
--use_vectorized_retrieval

疎行列・GPU版

  • entity→sentence、sentence→entityをPyTorch COO疎行列化
  • 全sentence類似度を先に一括計算
  • 重複排除、entity単位top-k、閾値処理はBFS版に合わせる
  • 大規模データで並列計算を活かしやすい

なぜdense検索だけでは難しいか

質問と所属passageが語彙的に遠い場合、直接のベクトル類似度だけでは順位が下がりやすい。

なぜrelation tripleが不要か

「提案した」「所属する」という意味は原文sentenceに残し、entity共起と文類似度で橋を選べる。

なぜPPRを最後に使うか

単一の局所一致ではなく、複数のactive entityから支持されるpassageを全体構造で上げられる。

8. 何が良くなったのか

精度面の意味

GraphRAG の「広く拾える」強みを残しつつ、relation noise を減らして relevance を改善した。

効率面の意味

relation extraction や graph summarization を避けたことで、indexing の重さを根本から削った。

実運用面の意味

大量コーパスや更新頻度の高いコーパスでも、コストを抑えて GraphRAG 的な検索を回しやすい。

9. 設計上の強みと、読むべき制約

「論文が主張する性質」と「このコードで確認できる挙動」を分けて整理します。

論文 実装 両方で確認
両方で確認

IndexingでLLMを呼ばない

NERとembeddingだけで索引を作る。LLMは取得passageから回答を生成する最終段階でのみ使われる。

両方で確認

Relation labelを保存しない

接続はentityの出現関係が中心。自然言語の関係意味はsentence・passage本文に保持する。

実装固有

隣接passage辺を追加する

0:... のような連番付きpassageを順番に接続し、文書の局所的な連続性もPPRへ入れる。

実装固有

属性質問向け補助boost

設定を有効化すると、where / born / founded等の語が質問とpassageで重なる場合に追加点を与えられる。

NER品質への依存 entityを落とすとグラフ経路自体が作れない。表記揺れも埋め込み対応だけで完全には解決しない。
Sentence共起はrelationの正しさを保証しない 同じ文に現れるだけで因果・所属・否定などの関係種別は区別しない。最終解釈は取得本文とLLMに委ねる。
パラメータが探索の広さを左右する iteration_threshold、max_iterations、top_k_sentence が小さすぎても大きすぎても、recallとnoiseのバランスが崩れる。
失敗時の退路が単純 seed entityが得られなければdense retrievalへ戻るため、グラフ検索だけに全面依存しない。

10. 初めての人向けの持ち帰り

LinearRAG は「GraphRAG をもっと賢くした」論文というより、 GraphRAG から壊れやすい部分だけを削って、残すべき構造だけを残した論文です。

向いているケース

  • 複数文書にまたがる multi-hop 質問を扱いたい
  • GraphRAG は使いたいが relation extraction の品質が信用しにくい
  • LLM トークンコストを indexing に使いたくない
  • 大きいコーパスを継続的に更新したい

注意して読むべき点

  • relation label 自体が必要な下流タスクには別評価が必要
  • entity extraction と sentence embedding の品質には依存する
  • 「relation を作らない」設計が常に最適とは限らない
  • ただし GraphRAG の現実的な代替案としてはかなり筋が良い
初学者向けに一言で言えば、「複雑な知識グラフを無理に作らず、原文を活かしたまま、必要なつながりだけを検索時に浮かび上がらせる」のが LinearRAG です。
